Phase 0 · Fondations

Lakehouse, Delta Lake
& Unity Catalog

Les concepts fondamentaux à maîtriser avant d'aborder les sections techniques de l'examen.

Pourquoi le Lakehouse existe

Avant Databricks, les entreprises choisissaient entre deux architectures imparfaites :

Data Lake : stockage cloud pas cher (S3, Azure Blob…), format ouvert, scalable. Mais aucune garantie de qualité, pas de transactions, requêtes lentes, impossible à gouverner.

Data Warehouse (Snowflake, BigQuery…) : performant, fiable, ACID. Mais cher, format propriétaire, pas adapté au ML ni aux données non structurées.

La plupart des entreprises maintenaient les deux en parallèle, avec des pipelines de synchronisation entre eux — coûteux et source d'incohérences.

À l'examen : Si la question décrit une entreprise avec des silos de données, deux systèmes séparés, ou de l'incohérence inter-équipes → la réponse est "lakehouse + Databricks".

Delta Lake : le moteur de fiabilité

Delta Lake est le format de stockage qui rend le lakehouse possible. Une table Delta = fichiers Parquet (données colonaires) + un dossier _delta_log (journal de chaque transaction).

Les 4 super-pouvoirs de Delta Lake

  • ACID : soit l'écriture est complète, soit elle n'a pas eu lieu. Jamais de données à moitié écrites.
  • Time travel : requêter les données telles qu'elles étaient à n'importe quelle version passée (VERSION AS OF, TIMESTAMP AS OF).
  • Schema enforcement : Delta refuse d'écrire des données dont le schéma ne correspond pas à la table.
  • OPTIMIZE / VACUUM : compactage des petits fichiers et nettoyage des fichiers obsolètes.
-- Time travel par version
SELECT * FROM orders VERSION AS OF 5;

-- Time travel par date
SELECT * FROM orders TIMESTAMP AS OF '2024-01-15';

-- Voir l'historique complet
DESCRIBE HISTORY orders;

-- Restaurer à une version antérieure
RESTORE TABLE orders TO VERSION AS OF 3;

-- Compacter les petits fichiers
OPTIMIZE orders ZORDER BY (customer_id);

-- Nettoyer les vieux fichiers (7 jours par défaut)
VACUUM orders RETAIN 168 HOURS;
Piège exam : Si tu exécutes VACUUM avec une rétention inférieure à 7 jours, tu perds la capacité de time travel pour les versions concernées. C'est un compromis explicite.

L'architecture Medallion : Bronze / Silver / Gold

Le pattern d'organisation des données dans un lakehouse. Pense à une raffinerie :

CoucheContenuRègle cléConsommateurs
BronzeDonnées brutes telles qu'elles arriventAppend-only, jamais modifiéesIngestion uniquement
SilverDonnées nettoyées, dédupliquées, validéesRègles qualité appliquéesData engineers
GoldAgrégations métier prêtes pour la BIModèle dimensionnel ou star schemaAnalysts, BI tools, data scientists
À l'examen : Les dashboards Tableau/Power BI doivent toujours interroger la couche Gold, jamais Silver ou Bronze. Silver est granulaire mais pas pré-agrégé.

Unity Catalog : la gouvernance centralisée

Unity Catalog (UC) répond à : qui a le droit de voir et faire quoi sur quelles données ?

Il impose une hiérarchie à 3 niveaux obligatoires :

Metastore
  └── Catalog  (ex: prod_catalog)
        └── Schema  (ex: sales)
              └── Table / View / Volume / Function

Pour accéder à une table, il faut des permissions à chaque niveau :

GRANT USE CATALOG ON CATALOG prod_catalog TO analysts;
GRANT USE SCHEMA ON SCHEMA prod_catalog.sales TO analysts;
GRANT SELECT ON TABLE prod_catalog.sales.orders TO analysts;
Piège exam classique : Un utilisateur a SELECT sur une table mais ne peut pas y accéder → il lui manque USE CATALOG et/ou USE SCHEMA. La chaîne doit être complète.

Les types de Compute

Databricks propose plusieurs types de compute selon l'usage :

TypeUsageCaractéristique clé
All-purpose clusterDéveloppement interactif, notebooksDémarré manuellement, persiste entre runs
Job clusterExécution de jobs automatisésDémarre et s'arrête automatiquement
SQL WarehouseRequêtes SQL analytiques, BIOptimisé pour SQL, auto-suspend
ServerlessJobs et pipelines sans gestion d'infraPas de configuration cluster, facturé à l'usage
À l'examen : Pour des requêtes SQL depuis Tableau → SQL Warehouse. Pour exécuter un pipeline en prod → Job cluster (plus économique car éphémère). Pour du développement interactif → All-purpose cluster.

Fonctionnalités des Notebooks

Les notebooks Databricks supportent plusieurs langages dans le même notebook :

-- SQL (par défaut dans une cellule SQL)
SELECT * FROM orders LIMIT 10;

# Python (prefixe %python ou cellule Python)
df = spark.read.table("orders")
display(df)

# Changer de langage dans une cellule
%sql
SELECT count(*) FROM orders;

%python
print("Hello")

dbutils est l'utilitaire de Databricks pour manipuler des fichiers, des secrets, et orchestrer des notebooks :

# Lister des fichiers
dbutils.fs.ls("dbfs:/data/")

# Passer des paramètres entre notebooks
result = dbutils.notebook.run("./child_notebook", timeout_seconds=60, arguments={"param": "value"})

# Accéder à un secret
secret = dbutils.secrets.get(scope="my_scope", key="my_key")

Serverless Compute

Avec le compute serverless, Databricks gère l'infrastructure à ta place. Avantages : pas de temps de démarrage de cluster, facturation à la seconde, pas de configuration. Utilisé pour les SQL Warehouses serverless, les jobs serverless et les pipelines Lakeflow serverless.

Important : Serverless ≠ gratuit. C'est un modèle de tarification différent (à l'usage plutôt qu'à l'heure de cluster), généralement plus cher à l'usage mais sans coût fixe.

Databricks Connect

Databricks Connect permet d'exécuter du code Spark depuis un IDE local (VS Code, PyCharm) sur un cluster Databricks distant. Utile pour le développement avec tests locaux.

Auto Loader vs COPY INTO

C'est le point de confusion numéro 1 à l'examen. Retiens cette règle simple :

Auto LoaderCOPY INTO
UsageIngestion continue en streamingChargement batch / one-shot
Nouveaux fichiersDétectés automatiquementNon, exécution manuelle
CheckpointRequis (obligatoire)Non requis
Schema locationRequis pour le streamingNon requis
SyntaxePySpark ou SQL (cloud_files)SQL uniquement

Syntaxe Auto Loader (PySpark)

df = spark.readStream \
    .format("cloudFiles") \
    .option("cloudFiles.format", "json") \
    .option("cloudFiles.schemaLocation", "/Volumes/cat/schema/ck1") \
    .option("cloudFiles.schemaEvolutionMode", "rescue") \
    .load("s3://data-lake/orders/")

df.writeStream \
    .trigger(availableNow=True)  # batch-like : traite tout puis s'arrête
    .format("delta") \
    .option("checkpointLocation", "/Volumes/cat/schema/write_ck") \
    .mode("append") \
    .table("orders_bronze")

Syntaxe COPY INTO (SQL)

COPY INTO catalog.schema.raw_data
FROM 's3://my-bucket/archive/2024-01/'
FILEFORMAT = JSON
FORMAT_OPTIONS ('inferSchema' = 'true')
COPY_OPTIONS ('mergeSchema' = 'true');

Modes de schéma evolution dans Auto Loader

  • addNewColumns : ajoute automatiquement les nouvelles colonnes (défaut)
  • rescue : garde le schéma existant, met les colonnes inconnues dans _rescued_data
  • failOnNewColumns : arrête le pipeline si une nouvelle colonne est détectée
  • none : ignore le schema stocké (rarement utilisé)
Piège exam : Sans cloudFiles.schemaLocation dans un pipeline streaming Auto Loader → le pipeline plante immédiatement. Ce paramètre est obligatoire en mode streaming.

Lakeflow Spark Declarative Pipelines

Anciennement "Delta Live Tables". Tu déclares tes tables en SQL ou Python, et Lakeflow gère l'orchestration, les dépendances, les retries et la qualité automatiquement.

Streaming Table vs Materialized View

STREAMING TABLEMATERIALIZED VIEW
RefreshIncrémental uniquementFull ou incrémental
UsageIngestion de nouvelles donnéesAgrégations et transformations
SourceExterne (cloud storage, kafka…)D'autres tables du pipeline
Keyword SQLCREATE STREAMING TABLECREATE MATERIALIZED VIEW
-- Bronze : streaming table depuis Auto Loader
CREATE STREAMING TABLE bronze_orders
COMMENT "Raw orders from cloud storage"
AS SELECT *
FROM cloud_files("s3://bucket/orders/", "csv",
  map("header", "true"));

-- Silver : materialized view avec contrainte qualité
CREATE MATERIALIZED VIEW silver_orders
CONSTRAINT valid_amount EXPECT (amount > 0) ON VIOLATION DROP ROW
AS SELECT order_id, customer_id, CAST(amount AS DECIMAL(10,2)) AS amount
FROM STREAM(live.bronze_orders);

-- Gold : agrégation
CREATE MATERIALIZED VIEW gold_daily_revenue
AS SELECT DATE(order_date) AS day, SUM(amount) AS revenue
FROM live.silver_orders
GROUP BY DATE(order_date);

CDC avec APPLY CHANGES

Change Data Capture : traitement d'un flux d'insertions, mises à jour et suppressions provenant d'un système source.

APPLY CHANGES INTO live.customers_silver
FROM STREAM(live.customers_cdc)
KEYS (customer_id)
SEQUENCE BY event_timestamp
WHEN MATCHED AND operation = 'DELETE' THEN DELETE
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

SCD Type 1 (défaut) : écrase l'ancienne valeur. SCD Type 2 (ajoute STORED AS SCD TYPE 2) : conserve l'historique en créant de nouvelles lignes avec colonnes __start_version, __end_version, __current.

À l'examen : SCD Type 2 est utilisé quand tu as besoin d'analyser l'évolution historique d'un enregistrement. SCD Type 1 quand tu n'as besoin que de l'état actuel.

Triggers dans les pipelines streaming

  • trigger(availableNow=True) : traite tous les fichiers disponibles une fois, puis s'arrête (comportement batch)
  • trigger(processingTime="5 minutes") : micro-batch toutes les 5 minutes
  • trigger(once=True) : même chose qu'availableNow (ancienne syntaxe)

DDL et DML Delta Lake

MERGE INTO — le pattern upsert

MERGE INTO target_table t
USING source_data s ON t.customer_id = s.customer_id
WHEN MATCHED AND s.operation = 'DELETE' THEN DELETE
WHEN MATCHED THEN UPDATE SET t.name = s.name, t.email = s.email
WHEN NOT MATCHED THEN INSERT (customer_id, name, email) VALUES (s.customer_id, s.name, s.email);
À l'examen : MERGE INTO est atomique. Si plusieurs WHEN MATCHED clauses existent, seule la première qui correspond est exécutée.

CREATE OR REPLACE TABLE (CRAS)

-- CRAS : atomique, préserve le delta log et l'historique
CREATE OR REPLACE TABLE summary_table AS
SELECT region, SUM(amount) as total FROM orders GROUP BY region;

-- ≠ DROP + CREATE (perd l'historique et n'est pas atomique)

INSERT INTO vs INSERT OVERWRITE

-- Ajoute des lignes (append)
INSERT INTO orders SELECT * FROM new_orders;

-- Remplace tout le contenu de la table
INSERT OVERWRITE orders SELECT * FROM corrected_orders;

PySpark DataFrame Aggregations

from pyspark.sql import functions as F

# Agrégations multiples en une seule passe
df.groupBy("category").agg(
    F.count("*").alias("total_orders"),
    F.sum("amount").alias("total_revenue"),
    F.avg("amount").alias("avg_value"),
    F.countDistinct("customer_id").alias("unique_customers")
).show()

Window Functions

from pyspark.sql.window import Window

window_spec = Window.partitionBy("region").orderBy(F.desc("revenue"))

df.withColumn("rank", F.row_number().over(window_spec)) \
  .withColumn("running_total", F.sum("revenue").over(
      Window.partitionBy("region").orderBy("date"))) \
  .filter(F.col("rank") <= 3)  # Top 3 par région
FonctionComportement avec ex-aequoExemple (1,1,3)
row_number()Rang strict, pas de doublons1, 2, 3
rank()Même rang pour les ex-aequo, gaps1, 1, 3
dense_rank()Même rang pour les ex-aequo, pas de gaps1, 1, 2

Optimisation des requêtes

Liquid Clustering (remplace le partitioning classique)

-- Activer Liquid Clustering sur une table
CREATE TABLE events (id BIGINT, event_date DATE, region STRING)
CLUSTER BY (event_date, region);

-- Sur une table existante
ALTER TABLE events CLUSTER BY (event_date, region);

-- OPTIMIZE pour réorganiser physiquement
OPTIMIZE events;

Liquid Clustering est plus flexible que le partitionnement classique : tu peux changer les colonnes de clustering sans recréer la table, et il gère mieux les colonnes haute cardinalité.

Predictive Optimization

Databricks peut automatiquement déclencher OPTIMIZE et VACUUM sur tes tables Unity Catalog sans que tu aies à le planifier. Active-le avec ALTER TABLE t SET TBLPROPERTIES ('delta.autoOptimize.optimizeWrite' = 'true').

Tables managées vs externes

Managed (gérée)External (externe)
StockageGéré par Unity CatalogChemin cloud que tu spécifies
DROP TABLESupprime données ET métadonnéesSupprime UNIQUEMENT les métadonnées
Usage typiqueTables internes au lakehouseDonnées existantes à l'extérieur
-- Table managée (Unity Catalog gère le stockage)
CREATE TABLE catalog.schema.orders (id INT, amount DECIMAL);

-- Table externe (tu spécifies l'emplacement)
CREATE TABLE catalog.schema.orders_external
LOCATION 's3://my-bucket/orders/';
Piège exam : DROP TABLE sur une table managée = destruction des données. Sur une table externe = seules les métadonnées sont supprimées, les données sur S3 restent intactes.

Databricks Workflows (Jobs)

Les Workflows permettent d'orchestrer des tâches (notebooks, scripts Python, pipelines Lakeflow) avec des dépendances, des planifications et une gestion des erreurs.

Repair & Rerun

Si un job multi-tâches échoue à l'étape 3/5, tu peux utiliser Repair Run pour reprendre depuis l'étape 3 sans re-exécuter les étapes 1 et 2 déjà réussies.

À l'examen : Repair Run = reprise depuis la tâche échouée. Re-run = tout recommencer depuis le début. Toujours préférer Repair Run pour les jobs longs.

Databricks Asset Bundles (DAB)

Les DAB permettent de gérer ton infrastructure Databricks (jobs, pipelines, clusters) comme du code (Infrastructure as Code), versionné avec Git et déployable en CI/CD.

# Structure d'un bundle
bundle:
  name: my-data-pipeline

resources:
  jobs:
    ingest_job:
      name: "Daily Ingestion - ${var.environment}"
      tasks:
        - task_key: ingest
          notebook_task:
            notebook_path: ./src/ingest
          new_cluster:
            spark_version: "14.3.x-scala2.12"
            node_type_id: "i3.xlarge"
            num_workers: 2
      schedule:
        quartz_cron_expression: "0 0 6 * * ?"
        timezone_id: "Europe/Paris"

variables:
  environment:
    default: dev
# Déploiement
databricks bundle deploy --target prod

# Exécution
databricks bundle run ingest_job

DAB vs déploiement traditionnel

DABDéploiement UI / API manuel
VersionningGit, PR, code reviewPas versionné
Multi-environnementsdev / staging / prod via variablesConfiguration manuelle par env
CI/CDNatif (GitHub Actions, Azure DevOps…)Scripts custom fragiles
ReproductibilitéGarantieNon garantie

Serverless Compute pour les Jobs

En mode serverless, pas besoin de configurer un cluster. Avantages : démarrage quasi-instantané (vs 2-5 min pour un cluster classique), pas de configuration. Idéal pour les jobs courts et fréquents.

Monitoring et Observabilité

Chaque pipeline Lakeflow génère un event log automatique accessible via :

SELECT * FROM event_log('pipeline_id')
WHERE level = 'ERROR'
ORDER BY timestamp DESC;

Le Spark UI permet d'analyser les performances : stages lents, skew de données, shuffles excessifs. Accès via le cluster > Spark UI dans l'interface Databricks.

Unity Catalog — Permissions détaillées

Rappel de la hiérarchie : Metastore → Catalog → Schema → Table/View/Volume/Function

Privilèges clés à retenir

PrivilègePermet deNiveau
USE CATALOGAccéder au catalog (obligatoire)Catalog
USE SCHEMAAccéder au schema (obligatoire)Schema
SELECTLire une table/vueTable/Schema/Catalog
MODIFYINSERT/UPDATE/DELETETable
CREATE TABLECréer des tables dans un schemaSchema
READ VOLUMELire depuis un volume UCVolume
WRITE VOLUMEÉcrire dans un volume UCVolume
ALL PRIVILEGESTous les droitsN'importe lequel
-- Donner accès complet à une équipe sur un schema
GRANT USE CATALOG ON CATALOG prod TO `data-team`;
GRANT USE SCHEMA ON SCHEMA prod.finance TO `data-team`;
GRANT SELECT, MODIFY ON SCHEMA prod.finance TO `data-team`;

-- Voir les permissions d'un objet
SHOW GRANTS ON TABLE prod.finance.transactions;

-- Révoquer
REVOKE SELECT ON TABLE prod.finance.transactions FROM `analyst-group`;

-- Transférer la propriété
ALTER TABLE prod.finance.transactions OWNER TO `platform-team`;
Héritage : GRANT SELECT ON CATALOG donne SELECT sur toutes les tables actuelles ET futures du catalog. L'héritage est automatique vers le bas.

Lineage dans Unity Catalog

Unity Catalog trace automatiquement la lineage au niveau colonne : d'où vient cette colonne ? Qui la lit ? Quelles tables en dépendent ?

Accessible via l'UI Databricks (Data Explorer > table > Lineage) ou via les system tables :

SELECT * FROM system.access.table_lineage
WHERE target_table_full_name = 'prod.finance.transactions';

Audit Logs

-- Qui a accédé à une table sensible ces 7 derniers jours ?
SELECT identity, action_name, event_time
FROM system.access.audit
WHERE request_params LIKE '%transactions%'
  AND event_time >= CURRENT_DATE() - INTERVAL 7 DAYS
ORDER BY event_time DESC;

Delta Sharing

Partage de données en direct sans copie, via un protocole ouvert. Le destinataire n'a pas besoin d'être sur Databricks.

D2D (Databricks to Databricks)D2O (Databricks to Open)
DestinataireSur DatabricksN'importe quel outil (Tableau, pandas…)
AuthUnity Catalog natifBearer token
AccèsRead-only (jamais write)Read-only

Modèle de coût : Le fournisseur paye le stockage. Le destinataire paye le compute de ses requêtes.

Piège exam : Delta Sharing est toujours read-only. Si le partenaire a besoin de modifier les données, il faut une autre solution (ETL, API). Et les partages ne permettent pas à l'extérieur d'écrire dans tes tables.

Lakehouse Federation

Permet de requêter des données dans des systèmes externes (PostgreSQL, MySQL, Snowflake, BigQuery…) directement depuis Databricks, sans les ingérer. Les données restent dans leur système source.

-- Créer une connexion vers PostgreSQL
CREATE CONNECTION pg_prod
TYPE POSTGRESQL
OPTIONS (host 'my-pg-host.com', port '5432', database 'sales');

-- Créer un catalog étranger (foreign catalog)
CREATE FOREIGN CATALOG pg_sales
USING CONNECTION pg_prod
OPTIONS (database 'sales_db');

-- Requêter comme n'importe quelle table
SELECT * FROM pg_sales.public.customers LIMIT 100;
